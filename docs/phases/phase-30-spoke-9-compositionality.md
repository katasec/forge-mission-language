## 1. Foundational concept

Compositionality is the ability to solve new, unfamiliar problems by combining known
building blocks in new ways. It is one of the most fundamental properties of human
intelligence — we can handle novel situations because we can reassemble what we already
know.

LLMs struggle with this. Research shows that as tasks require more steps of composition
(apply rule A, then rule B, then rule C in a new combination), LLM accuracy degrades
sharply — even when the model handles each rule correctly in isolation. The model
pattern-matches against training data; it cannot reliably *compose* rules it has
not seen combined before.

The neurosymbolic solution: decompose complex novel tasks into known sub-tasks, assign
each sub-task to a specialised expert, then compose the results. The composition
structure is explicit and reliable (symbolic). The individual expert outputs are
generated by LLMs (neural). Neither alone handles the novel task well; together, they do.

Marcus: *"Embeddings aren't enough. Sooner or later you are going to need true compositionality."*
BAIR (Zaharia et al.): *"State-of-the-art AI results are increasingly obtained by compound
systems with multiple components, not just monolithic models."*

## 3. How MCL demonstrates this

```
TaskDecomposer
-> parallel {
    PrimitiveExpertA
    PrimitiveExpertB
    PrimitiveExpertC
}
-> Composer
```

- `TaskDecomposer` — LLM expert; receives a novel complex task; breaks it into
  sub-tasks matching known expert capabilities; writes sub-task assignments to context
- `PrimitiveExpertA/B/C` — specialised LLM experts, each handling one well-defined
  primitive (e.g. A: factual lookup, B: numerical calculation, C: sentiment/tone
  assessment); run independently in parallel
- `Composer` — LLM expert; reads all three primitive outputs; assembles them into a
  coherent final answer for the novel task

**Demo task (to decide):** A task that is genuinely novel — not something the LLM
would handle well in one shot — but is decomposable into primitives each expert
handles reliably. Example: "Evaluate whether this investment memo is sound" decomposed
into (1) check the numbers are consistent, (2) assess the market context claims,
(3) evaluate the risk language — each a known primitive, combined into a novel whole.

**What MCL adds:** the composition structure is explicit in the `.mcl` file. Anyone
reading it can see exactly how the novel task is decomposed. The `TaskDecomposer`
makes the routing transparent — not buried in a routing weight or an opaque agent loop.
Changing how a task is decomposed means editing the mission file, not retraining a model.

## 4. Why this is normally hard

Without MCL, compositional task routing requires:

- An LLM call to decompose the task (and code to parse its output into a routing decision)
- Dynamic dispatch to the right expert based on the decomposition
- Parallel execution orchestration (async, thread pools, or agent frameworks)
- Result aggregation: collecting outputs from parallel experts and passing them to the
  composer
- Error handling if any sub-task fails

In practice, most teams implement this with a monolithic prompt ("please do A, B, and C")
and hope the model handles the composition. The research shows it often doesn't.

The alternative — building proper compositional routing in LangChain, AutoGen, or a
custom agent framework — requires significant engineering: understanding async execution,
dynamic prompt construction, output parsing, and agent state management.

The person who knows which primitives to apply to a novel problem in their domain —
the analyst, the doctor, the engineer — cannot build this system today. They need a
developer, who needs to understand both the domain decomposition and the orchestration
framework.

With MCL, the domain expert writes the decomposition into the `TaskDecomposer` system
prompt and names the experts in the `parallel {}` block. The composition is declared,
not coded.
