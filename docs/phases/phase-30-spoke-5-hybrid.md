# Phase 30 Spoke 5 — Hybrid LLM + Classical ML

> **Status: Brainstorm**
> **Depends on:** Phase 22b (ONNX) — requires `kind: onnx` and `kind: json_extract`
> **Papers:**
> - [Composition of Experts / CoE (2024)](https://arxiv.org/pdf/2412.01868)
> - [Towards Generalized Routing: Model and Agent Orchestration (2024)](https://arxiv.org/html/2509.07571v1)

---

## Paper summary

**CoE:** Route input prompts to one of a set of modular experts using a routing function.
Compose expert outputs rather than running a single monolithic model. Key result: modular
composition of medium-sized specialised models can outperform larger monolithic models.

**Routing / Orchestration:** The orchestration layer — how inputs are routed to the right
expert — is the critical component. The paper explores generalised routing across heterogeneous
model types, not just homogeneous LLM agents.

**Combined thesis:** Heterogeneous expert composition (LLM + classical ML) is a natural
extension of the CoE pattern. The pipeline does not need to be all-LLM to benefit from
composition.

---

## MCL expression

```
LLMAnalyst → FeatureExtractor (json_extract) → Scorer (onnx) → LLMInterpreter
```

- `LLMAnalyst` — LLM expert; processes unstructured input, produces structured analysis
- `FeatureExtractor` — `kind: json_extract`; bridges LLM output (JSON string) into typed
  numeric context keys (`feature_a`, `feature_b`, `feature_c`)
- `Scorer` — `kind: onnx`; reads numeric features, runs a trained ML model, writes a score
  back to context
- `LLMInterpreter` — LLM expert; reads the score and the original analysis; produces
  a human-readable interpretation and recommendation

This is the text-quality-demo pattern generalised as a concept mission. The mission file
makes the LLM ↔ classical ML handoff explicit and legible.

**What MCL adds:** the pipeline treats an ONNX model as a first-class expert — not a
preprocessing step or external call bolted on outside the language. The composition is
declared in MCL syntax and the handoff is handled by the runtime.

## Demo task (to decide)

Options under consideration:
- **Sentiment + toxicity scoring** — LLM analyses text sentiment → json_extract → ONNX
  toxicity classifier → LLM generates moderation recommendation
- **Document quality** — LLM extracts quality signals from text → json_extract → ONNX
  quality scorer → LLM writes improvement notes (close to text-quality-demo but more general)
- **Risk assessment** — LLM extracts risk signals from a description → json_extract →
  ONNX risk model → LLM writes a risk summary

Preference for a task where the classical ML model adds something the LLM alone cannot
reliably provide — calibrated numeric scoring, not just qualitative assessment.

## What success looks like

A reader can see exactly where the LLM hands off to classical ML and back again — in
the mission file and in the README. The ONNX model adds a calibrated signal that the
LLM alone would produce inconsistently. The pipeline is runnable with the provided
`generate_model.py`.
